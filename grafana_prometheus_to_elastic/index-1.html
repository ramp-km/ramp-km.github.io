No Rewrites

Run existing PromQL natively

Your Prometheus dashboards, alert rules and scrape configs keep working — PromQL runs directly in Elastic, and the same data is queryable in ES|QL.

PromQL — unchanged
# existing query, still valid
rate(http_requests_total{status="5xx"}[5m])
  / rate(http_requests_total[5m])
Same data via ES|QL
FROM metrics-*
| WHERE status == "5xx"
| STATS error_rate = RATE(count) BY service
| SORT error_rate DESC

Same result, one less silo — correlate metrics with logs and traces in the same query language.

https://www.elastic.co/docs/reference/query-languages/esql/commands/promql

Migration

Strategies, best practices & common pitfalls

Do this
  • Inventory existing PromQL queries and alert rules before migrating
  • Migrate read paths first, validate against production dashboards
  • Run Prometheus and Elastic side-by-side during cutover
  • Right-size retention using the new storage-efficiency gains
Avoid this
  • Migrating alerting before dashboards are validated
  • Assuming every label/cardinality pattern needs to carry over as-is
  • Skipping a rollback plan during cutover
  • Treating migration as a one-shot, big-bang event
